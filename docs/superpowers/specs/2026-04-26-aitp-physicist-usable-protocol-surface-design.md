# AITP Physicist-Usable Protocol Surface Design

## Status

Recovered and rewritten 2026-04-26 from original 2026-04-18 intent plus
post-recovery codebase analysis. Replaces the lost spec from session 1ae58224.

## Problem Statement

### P1: Active contract semantics diluted by runtime generalization

The protocol defines `question_contract.md` with specific fields — bounded question,
scope boundaries, target quantities. But the runtime treats it identically to
every other L1 artifact: check that frontmatter fields are non-empty, check that
headings exist. The gate passes when "something is filled in," not when the
question is genuinely well-posed.

**Consequence**: Topics advance to L3 with vague questions. Downstream derivation
inherits this ambiguity. The contract has no teeth.

**Fix**: Add a semantic validity gate to `question_contract.md` that checks for
structural completeness of the question itself, not just field presence.

### P2: Human-facing next action has no physics content

`aitp_get_execution_brief` returns "edit flagged artifact" or "check L1/gate_status."
The physicist is told to maintain artifacts, not to think about physics. The
"next step" has degenerated into queue maintenance.

**Consequence**: AITP feels like a bureaucracy, not a research collaborator.
Physicists disengage.

**Fix**: Generate physics-contentful next actions. Instead of "fill in
question_contract.md," say "decide whether the AdS radius R sets the energy
scale, or the string coupling g_s does — this choice changes which EFT tower
layer your question belongs to."

### P3: L1 completeness gate existed but lacked mechanical enforcement

The protocol always required source coverage, but it relied on the human or AI
to notice gaps. There was no mechanical check that every section in a paper's
TOC had been extracted or explicitly deferred.

**Fix**: Implemented in the recovered session 63654680 edits (L1 TOC workflow).
Already deployed.

### P4: L4 adversarial review is procedural, not adversarial

The L4 review gate requires a review artifact, but the review itself can be
perfunctory. There is no requirement that the review identify a specific
weakness or propose a counterargument.

**Consequence**: Candidates pass L4 without genuine adversarial scrutiny.

**Fix**: Require at least one specific counterargument or limitation-flag in
every L4 review. Make this mechanically checkable.

### P5: L2 knowledge graph is empty

Architecture exists (node types, edge types, EFT towers, trust evolution) but
no physics content. The knowledge graph cannot serve its purpose if zero nodes
exist.

**Fix**: Populate L2 with foundational physics concepts. This is Phase 2 of
the broader roadmap (separate from this spec).

## Design

### Workstream A: Question contract semantic validity (P1)

**What**: Add a structural check to `evaluate_l1_stage` that validates the
bounded question is genuinely well-posed.

**Checklist** (any failure blocks L1→L3 advancement):
- [ ] `bounded_question` contains at least one of: "what is", "how does",
  "why is", "under what conditions", "derive", "compute", "estimate", "prove"
- [ ] `scope_boundaries` identifies at least one explicit exclusion ("this
  question does NOT ask about X")
- [ ] `target_quantities` names at least one physically measurable or calculable
  quantity with units or dimensionless characterization
- [ ] `## Non-Success Conditions` is filled and at least one specific failure
  mode is described (not "we don't find anything")

**Implementation**: Extend `evaluate_l1_stage` in `brain/state_model.py`. Add a
semantic check function `_check_question_semantic_validity` called after the
existing field-presence checks. ~40 lines of Python.

**Acceptance test**: Create a topic with a vague question ("study quantum gravity").
Verify gate_status == "blocked_coverage_incomplete" with specific missing
requirement listing which check failed.

### Workstream B: Physics-contentful next actions (P2)

**What**: `aitp_get_execution_brief` returns a `next_action` field that, when
the topic has a well-formed question, includes a physics-specific prompt derived
from the current gate context and domain.

**Design**:
- If gate_status is not ready: return the specific gate fix needed (existing behavior)
- If gate_status is ready: generate a physics-contentful prompt using the
  question contract + domain context
- The prompt is generated by template, not LLM: fill slots from the contract

**Template** (example for L1→L3 transition):
```
L1 is complete. Before deriving, clarify:
- Your question: {bounded_question}
- Energy scale: {from domain manifest or TOC}
- Key observable: {target_quantities}
- Explicitly excluded: {scope_boundaries}
Proceed to L3 derivation, or refine the question first?
```

**Implementation**: Add `_generate_physics_next_action()` to
`brain/state_model.py`, call it from `aitp_get_execution_brief` in
`brain/mcp_server.py`. ~60 lines.

**Acceptance test**: Bootstrap a topic with a well-formed question, advance to
L1 ready. Call `aitp_get_execution_brief`. Verify `next_action` contains the
bounded question text and target quantities, not just "proceed to L3."

### Workstream C: L4 adversarial hardening (P4)

**What**: Require at least one counterargument in every L4 review.

**Checklist** for L4 gate (added to `evaluate_l4_stage`):
- [ ] Review body contains `## Counterargument` or `## Limitations Identified`
- [ ] At least one specific claim is challenged with a reason (not "more tests
  needed" — must name a specific assumption, regime boundary, or competing
  interpretation)

**Implementation**: Extend `evaluate_l4_stage` in `brain/state_model.py` to
parse review files and check for counterargument structure. ~35 lines.

**Acceptance test**: Submit a candidate with a review that has no
counterargument section. Verify gate_status != "ready" with reason mentioning
missing counterargument.

### Workstream D: Domain-manifest-driven energy scale routing (P2 extension)

**What**: When a topic's domain is known (from question contract or source TOC),
automatically suggest the relevant EFT tower layer and energy scale.

**Implementation**: Extend `aitp_get_execution_brief` to cross-reference the
domain taxonomy with existing L2 EFT towers. If a matching tower exists, include
energy-scale context in the physics-contentful next action. ~30 lines.

**Prerequisite**: Domain taxonomy must exist (Phase 2 work). This workstream is
`deferred` until after Phase 2.

## Workstream Ordering

```
Phase 1 (this spec, immediate):
  A → Question contract semantic validity
  B → Physics-contentful next actions
  C → L4 adversarial hardening

Phase 2 (separate spec):
  L2 domain taxonomy + content population

Phase 3 (deferred):
  D → Domain-manifest-driven energy scale routing
```

A, B, C are independent and can be implemented in parallel. Each is ~35-60 lines
of Python with mechanical acceptance tests.

## Non-Goals (explicitly removed from this round)

- **Full L2 content population**: Too large for this spec. Separate Phase 2 work.
- **Lean bridge integration**: Deferred until L2 has content worth verifying.
- **Multi-topic cross-referencing**: Requires populated L2 first.
- **GUI / dashboard redesign**: Valuable but not protocol-core.
- **Auto-formalization of derivations**: Premature before L2 content exists.

## Acceptance Criteria

| Workstream | Test | Criterion |
|-----------|------|-----------|
| A | `test_state_model.py::test_question_semantic_gate_blocks_vague` | Gate blocks when question lacks exclusion or measurable target |
| A | `test_state_model.py::test_question_semantic_gate_passes_well_posed` | Gate passes for well-formed question |
| B | `test_mcp_server.py::test_execution_brief_has_physics_content` | Brief includes question text and target quantities |
| B | `test_mcp_server.py::test_execution_brief_falls_back_for_vague_question` | Falls back to gate-fix message when question is vague |
| C | `test_state_model.py::test_l4_gate_blocks_missing_counterargument` | Gate blocks when review has no counterargument |
| C | `test_state_model.py::test_l4_gate_passes_with_counterargument` | Gate passes when counterargument is present |

## Risks

- **Over-gating L1**: If the question semantic check is too strict, topics get
  stuck at L1. Mitigation: checks are structural (has exclusion statement, has
  measurable quantity), not qualitative (is this a "good" question?).
- **Template fatigue**: Physics-contentful next actions from templates may feel
  formulaic. Mitigation: start with templates, iterate based on real usage.
- **L4 adversarial check evasion**: Reviews can include a trivial counterargument
  ("assumes flat spacetime"). Mitigation: the check requires naming a specific
  claim from the candidate, which forces reading the candidate.
- **Scope creep into Phase 2**: Workstream D is explicitly deferred. If L2
  content arrives sooner than expected, D can be unblocked.

## Summary

Three concrete, independently testable workstreams that directly address the
four highest-severity issues in the current AITP protocol surface. Each is
scoped to 35-60 lines of Python. Combined, they transform the protocol from
"artifact-checking bureaucracy" to "physics-contentful research collaborator."

The spec avoids roadmap-manifesto language: every workstream has a specific
implementation file, line-count estimate, and mechanically verifiable acceptance
test.
